# 01 · The Struct Skeleton (Python port)

Python port of `week1_baseline/ruby/01_struct_skeleton`. Same data shapes,
same behaviour — see that step's README for the full design rationale. This
file only covers the Python-specific API and setup.

Like the Ruby step, this defines the three core data structures:
- `boukensha.Tool`
- `boukensha.Message`
- `boukensha.Context`

Ruby represents these as `Struct`s for lightweight, readable data
containers. The Python equivalent is `@dataclass` for `Tool` and `Message`
(mutable field containers with a custom `__str__`); `Context` stays a plain
class since it owns mutable state (`messages`, `tools`) and behavior
(`register_tool`, `add_message`).

## Setup

Uses the same shared repo-root `.venv` as `python/00_config` — see that
step's README for the one-time setup. To run this step specifically:

```bash
.venv/bin/pip install -r week1_baseline/python/01_struct_skeleton/requirements.txt
.venv/bin/pip install -e week1_baseline/python/01_struct_skeleton
```

This step is a self-contained copy of the `boukensha` package (mirroring
Ruby's per-step-folder duplication — `config.py`, `tasks/base.py`, and
`tasks/player.py` are copied forward unchanged from `python/00_config`,
minus the `PROMPTS_DIR` constant, which this step's `Config` no longer
needs). Installing it editable repoints `import boukensha` at this step's
copy.

## Data Structures

### `boukensha.Tool`
| Field | Description |
|---|---|
| `name` | The name of the tool |
| `description` | Shown to the agent so it knows when to invoke the tool |
| `parameters` | The arguments that need to be passed in |
| `block` | The actual code that runs when the tool is called (a `Callable`) |

**Examples:**
```
<Tool name=move description="Move the player in a direction (north, south, east, west, up, down)" params=['direction']>
<Tool name=attack description="Attack a target in the current room" params=['target']>
<Tool name=look description="Look around the current room" params=[]>
<Tool name=say description="Say something out loud to others in the room" params=['message']>
```

### `boukensha.Message`
A single unit of conversation between the user and the agent.

| Field | Description |
|---|---|
| `role` | Who is speaking — user, assistant, or tool_result |
| `content` | The text generated by the agent or provided by the user |
| `tool_use_id` | Links a tool result back to the specific tool call that requested it. The API requires this pairing to be exact. |

**Examples:**
```
<Message role=user content=Explore north and tell me what you find....>
<Message role=assistant content=Sure, let me head north and take a look....>
<Message role=tool_result [toolu_01X] content=You move north into a torch-lit corridor....>
```

### `boukensha.Context`
Holds everything BOUKENSHA needs to make an API call. Nothing lives outside of this.

| Field | Description |
|---|---|
| `system` | Tells the agent what its job is. Sent as a separate field in the API call, never inside the messages array. |
| `messages` | The full conversation history, replayed on every turn |
| `tools` | The registered tools the agent is allowed to invoke |

```
<Context task=player turns=2 tools=1>
```

## Ruby → Python notes

- Ruby's `Struct#to_s` inclusive ranges (`description.to_s[0..40]`,
  `content.to_s[0..60]`) become Python's exclusive-end slices
  (`description[:41]`, `content[:61]`) — same substring length.
- Ruby prints hash symbol keys as `params=[:direction]`; Python prints
  `params=['direction']` — same data, different literal repr. Same class of
  cosmetic difference as `#<...>` vs `<...>` in step 00.
- Ruby's `Config#tasks`/`#dig` accept both symbol and string keys; the
  Python `Config` only takes plain `str` (no symbol type in Python).
